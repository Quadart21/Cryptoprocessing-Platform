# Полное ТЗ: SaaS CryptoProcessing Platform

## 1. Назначение проекта

Разработать multi-tenant SaaS-платформу для управления криптоплатежами, магазинами и сопутствующей операционной деятельностью клиентов через единый web-интерфейс.

Платформа должна:
- выступать абстрактным слоем над Crypto-Cash API без прямого доступа клиентов к Crypto-Cash;
- позволять супер-админу централизованно создавать и управлять аккаунтами клиентов;
- предоставлять каждому клиенту изолированный кабинет со своими проектами, платежами, бухгалтерией, товарами, сотрудниками и настройками;
- в перспективе поддерживать no-code создание Telegram-бота и торгового сайта на стороне клиента.

## 2. Ключевая бизнес-логика

### 2.1 Основная модель

Платформа работает по модели tenant-based SaaS:
- один клиент = один tenant;
- данные tenants полностью изолированы на уровне БД и backend-логики;
- супер-админ видит все tenants, все транзакции, все настройки и все системные журналы;
- клиент видит только свои данные и данные сотрудников своего tenant.

### 2.2 Базовый сценарий работы

1. Клиент самостоятельно регистрируется на платформе и подает заявку на проект.
2. При регистрации создаются пользователь, tenant-заявка и проект в статусе `pending_review`.
3. До одобрения клиент видит только ограниченный onboarding-кабинет со статусом заявки.
4. Супер-админ проверяет заявку, домен и параметры проекта, после чего одобряет или отклоняет ее.
5. После одобрения активируются tenant, проект, API-ключи и рабочий кабинет клиента.
6. Через API или кабинет клиент создает инвойсы на оплату.
7. Backend платформы вызывает Crypto-Cash API от своего имени.
8. Статусы платежей приходят по webhook от Crypto-Cash в платформу.
9. Платформа обновляет внутренние сущности, проводки, баланс и отправляет webhook клиенту.
10. Клиент видит платежи, баланс, комиссии, выгрузки и отчеты в своем кабинете.

## 3. Роли и права доступа

### 3.1 Роли

- `superadmin`:
  полный доступ ко всей платформе, настройкам, tenants, платежам, комиссиям, логам, интеграциям и управлению пользователями.
- `tenant_owner`:
  владелец tenant, управляет своим кабинетом, API-ключами, проектами, товарами, сотрудниками, настройками магазина и отчетами.
- `tenant_admin`:
  расширенное управление в пределах tenant без доступа к глобальным системным настройкам.
- `tenant_manager`:
  операционная роль, работа с заказами, товарами, клиентскими сущностями, статусами и отчетами в пределах выданных прав.
- `tenant_accountant`:
  доступ к бухгалтерии, балансам, экспортам, отчетам, транзакциям и начислениям.
- `tenant_employee`:
  ограниченный доступ по назначенным разделам.

### 3.2 Принципы доступа

- Публичная регистрация клиента разрешена только в формате подачи заявки на проект.
- Полный доступ к рабочему кабинету и платежным функциям открывается только после одобрения супер-админом.
- Создание сотрудников выполняется владельцем tenant или админом tenant после одобрения проекта.
- Все запросы клиента ограничиваются его `tenant_id`.
- На уровне PostgreSQL используется RLS.

## 5. Функциональные требования

## 5.1 Аутентификация и доступ

### Требования

- Вход по email/паролю.
- Установка пароля по invite-ссылке.
- JWT access token + refresh token.
- Хеширование паролей через bcrypt/argon2.
- Возможность включения 2FA для супер-админа.
- Принудительный logout при блокировке пользователя.
- История входов: дата, IP, user-agent, результат попытки.

### Сценарии

- Клиент сам регистрируется через публичную форму.
- Система создает аккаунт и tenant/project в статусе ожидания.
- После подтверждения email пользователь попадает в ограниченный onboarding-кабинет.
- После одобрения супер-админом пользователь получает полный доступ к tenant-кабинету.

## 5.2 Онбординг клиентов

### Требования

- Публичная форма регистрации `/register`:
  - название компании;
  - имя владельца;
  - email;
  - пароль;
  - домен проекта;
  - описание проекта;
  - Telegram/контакт;
  - согласие с правилами.
- При регистрации автоматически создаются:
  - `tenant`;
  - первичный пользователь `tenant_owner`;
  - начальный проект;
  - заявка на модерацию;
  - audit trail события регистрации.
- До одобрения API-ключи неактивны или не выдаются клиенту.
- После одобрения автоматически активируются:
  - tenant;
  - проект;
  - API-ключи;
  - базовые настройки бухгалтерии.
- Статусы tenant:
  - `pending_review`;
  - `approved`;
  - `rejected`;
  - `suspended`;
  - `archived`.
- Статусы проекта:
  - `pending_review`;
  - `active`;
  - `rejected`;
  - `disabled`.

## 5.11 Webhook

- `/internal/webhook/crypto-cash`

Требования:

- журналирование заголовков и payload;
- защита от повторной доставки;

- подпись webhook со стороны платформы;
- журнал попыток доставки;
- переключатель активности;
- тестовая отправка webhook из кабинета.

### 7.1 Супер-бухгалтерия для супер-админа

Супер-админу нужен отдельный расширенный бухгалтерский контур по всей платформе, в котором он видит не только данные конкретного клиента, а полную сводную картину бизнеса.

#### Глобальные показатели платформы

Супер-админ должен видеть:
- общее количество созданных инвойсов;
- общее количество оплаченных инвойсов;
- общее количество подтвержденных инвойсов;
- общее количество отмененных и истекших инвойсов;
- общую сумму всех созданных инвойсов;
- общую сумму успешно оплаченных инвойсов;
- общую сумму подтвержденных поступлений;
- общую сумму комиссий платформы;
- общую сумму комиссий провайдера;
- чистую сумму после вычета комиссий;
- средний чек по платформе;
- оборот за день;
- оборот за неделю;
- оборот за месяц;
- оборот за произвольный период;
- количество новых клиентов за период;
- количество активных клиентов;
- количество клиентов без платежей;
- количество клиентов с просроченными/проблемными webhook.

#### Аналитика по клиентам

Супер-админ должен иметь возможность смотреть:
- сколько инвойсов создал каждый tenant;
- на какую общую сумму созданы инвойсы по каждому tenant;
- сколько оплачено по каждому tenant;
- сколько подтверждено по каждому tenant;
- сколько комиссий принес каждый tenant;
- средний чек по каждому tenant;
- последний платеж tenant;
- последний созданный инвойс tenant;
- долю tenant в общем обороте платформы;
- рейтинг top tenants по обороту;
- рейтинг top tenants по количеству инвойсов.

#### Аналитика по статусам и качеству платежного потока

Супер-админ должен видеть:
- конверсию из `created` в `paid`;
- конверсию из `paid` в `confirmed`;
- долю неуспешных, отмененных и истекших инвойсов;
- распределение инвойсов по статусам;
- количество зависших платежей;
- количество инвойсов без webhook-подтверждения;
- количество повторных попыток webhook;
- проблемные проекты и tenants с аномально высоким числом неуспешных платежей.

#### Срезы и фильтры

Все бухгалтерские и аналитические данные должны фильтроваться по:
- периоду;
- tenant;
- проекту;
- валюте;
- сети;

## 14. Бизнес-правила

- Нельзя удалить tenant с транзакционной историей, только архивировать.
- Нельзя физически удалять подтвержденные транзакции и инвойсы.
- Регенерация API-ключа инвалидирует старый секрет.
- До одобрения tenant:
  - клиент не может создавать инвойсы;
  - клиент не видит боевые API-ключи;
  - клиент работает только в onboarding-режиме.
- При блокировке tenant:
  - запрещается новый логин;
  - останавливается создание новых инвойсов;
  - данные остаются доступными супер-админу.
- Все финансовые изменения должны сопровождаться ledger entry.
- Все изменения статусов платежа должны быть трассируемыми.

## 14.1 Единый баланс Crypto-Cash и внутренний учет платформы

### Принцип

- У Crypto-Cash используется один общий провайдерский баланс на всю платформу.
- Клиенты платформы не имеют отдельных реальных балансов в Crypto-Cash.
- Для каждого tenant платформа обязана вести собственный внутренний баланс и ledger.
- Именно внутренний ledger платформы является источником истины для клиентского кабинета, админки, заявок на вывод и сверки.

### Модель учета

- Каждый успешный платеж сначала попадает в общий баланс Crypto-Cash.
- Затем платформа отражает этот платеж во внутреннем учете конкретного tenant.
- Для tenant должны считаться как минимум:
  - `gross_amount` — полная сумма платежа;
  - `provider_fee_amount` — комиссия Crypto-Cash;
  - `platform_markup_amount` — ваша наценка;
  - `net_amount` — сумма, доступная клиенту после удержаний;
  - `pending_amount` — средства в ожидании подтверждения/разблокировки;
  - `available_amount` — средства, доступные к выводу;
  - `locked_amount` — средства, зарезервированные под заявку на вывод;
  - `withdrawn_amount` — уже выплаченные клиенту средства.

### Комиссии

- Базовая комиссия провайдера Crypto-Cash: `0.2%` от каждого проведенного платежа.
- Платформа должна поддерживать собственную наценку поверх комиссии провайдера.
- Наценка должна поддерживаться в двух режимах:
  - глобальная наценка по умолчанию для всех tenants;
  - индивидуальная наценка для конкретного tenant.
- Если для tenant задана индивидуальная наценка, она должна иметь приоритет над глобальной.
- В админке должно быть видно:
  - комиссия провайдера;
  - ваша наценка;
  - чистая сумма клиента;
  - общая прибыль платформы.

### Выплаты клиентам

- Клиент не выводит средства напрямую из Crypto-Cash.
- Клиент создает заявку на выплату внутри вашей платформы.
- Заявка на выплату должна содержать:
  - tenant;
  - проект;
  - валюту;
  - сеть;
  - адрес назначения;
  - сумму запроса;
  - статус;
  - комментарий модерации;
  - external payout id провайдера при фактической отправке.
- Статусы заявки на выплату:
  - `pending_review`;
  - `approved`;
  - `processing`;
  - `paid`;
  - `rejected`;
  - `cancelled`;
  - `failed`.
- При создании заявки сумма должна переводиться из `available` в `locked`.
- При одобрении и фактической выплате сумма должна переводиться из `locked` в `withdrawn`.
- При отклонении или отмене сумма должна возвращаться из `locked` в `available`.

### Ledger

- Любое финансовое изменение должно записываться отдельной ledger-проводкой.
- Ledger должен покрывать:
  - начисление валового дохода;
  - удержание комиссии Crypto-Cash;
  - удержание вашей наценки;
  - перевод из `pending` в `available`;
  - резервирование под payout;
  - списание payout;
  - возвраты и корректировки;
  - ручные корректировки супер-админа.
- Все ledger-проводки должны быть привязуемы к invoice, transaction или payout request.

### Супер-админская бухгалтерия

- Супер-админ должен видеть не только агрегаты по инвойсам, но и полноценную картину обязательств платформы:
  - общий баланс Crypto-Cash;
  - сумму обязательств перед всеми tenants;
  - сколько находится в `pending`;
  - сколько доступно клиентам к выводу;
  - сколько зарезервировано в payout;
  - сколько уже выплачено;
  - сколько составляет доход платформы по наценке;
  - сколько удержано в пользу провайдера;
  - разницу между балансом провайдера и суммой внутренних обязательств.

### Сверка

- В системе должен существовать механизм сверки:
  - провайдерский общий баланс Crypto-Cash;
  - сумма всех внутренних клиентских балансов;
  - сумма pending/locked/withdrawn;
  - накопленные комиссии платформы и провайдера.
- Любая расхожесть должна фиксироваться в админке как reconciliation issue.

Итоговая версия продукта должна быть не просто криптоплатежным шлюзом, а администрируемой SaaS-платформой, где супер-админ централизованно создает и контролирует клиентские аккаунты, а каждый клиент получает собственную изолированную экосистему:
- прием криптоплатежей;
- проекты и API;
- бухгалтерия;
- товары и заказы;
- сотрудники и роли;
- в дальнейшем Telegram-бот и storefront без программирования.

Это ТЗ можно использовать как базовый master-document для:
- старта разработки;
- декомпозиции в backlog;
- оценки подрядчиком;
- оформления в Notion, ClickUp или Jira.
